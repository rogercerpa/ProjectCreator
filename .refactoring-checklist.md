# Refactoring Safety Checklist

## Pre-Refactoring Setup

### 1. Backup Current State
- [ ] Create backup branch: `git checkout -b backup/pre-refactor-$(date +%Y%m%d)`
- [ ] Tag current version: `git tag v5.0.93-pre-refactor`
- [ ] Export current package-lock.json: `cp package-lock.json package-lock.backup.json`
- [ ] Take screenshot of working app
- [ ] Document current known issues

### 2. Establish Baseline
- [ ] Run all tests: `npm test`
- [ ] Record test coverage: `npm run test:coverage`
- [ ] Check app startup time
- [ ] Profile memory usage
- [ ] List all working features

### 3. Setup Safety Tools
- [ ] Install git hooks (husky)
- [ ] Setup ESLint
- [ ] Configure Prettier
- [ ] Enable TypeScript checking (incremental)
- [ ] Setup automated tests on commit

---

## During Refactoring

### For Each Change

#### Before Starting
- [ ] Create feature branch from develop
- [ ] Write test for current behavior
- [ ] Document what you're changing and why
- [ ] Estimate risk level (low/medium/high)

#### During Development
- [ ] Make small, incremental changes
- [ ] Commit after each logical step
- [ ] Run tests frequently
- [ ] Keep app running and test manually

#### Before Committing
- [ ] All tests pass: `npm test`
- [ ] No linter errors: `npm run lint`
- [ ] App starts: `npm start`
- [ ] Core features work:
  - [ ] Create new project
  - [ ] Load existing project
  - [ ] Save project
  - [ ] Open settings
  - [ ] Agency directory loads
- [ ] No console errors
- [ ] No new warnings

#### After Merge
- [ ] Delete feature branch
- [ ] Update documentation if needed
- [ ] Notify team of changes

## Risk Levels

### 🟢 Low Risk
- Adding comments
- Renaming variables (within function)
- Extracting pure functions
- Adding tests

### 🟡 Medium Risk
- Extracting modules
- Moving code between files
- Changing data structures
- Updating dependencies

### 🔴 High Risk
- Changing IPC communication
- Modifying core services
- Changing persistence layer
- Updating main process logic

## Emergency Procedures

### If App Won't Start
```bash
# 1. Check for syntax errors
npm run build

# 2. Clear cache
rm -rf node_modules dist .webpack_cache
npm install
npm run build

# 3. Rollback to last working commit
git reset --hard HEAD~1
npm run build
npm start
```

### If Tests Fail
```bash
# 1. Run specific test to see error
npm test -- --testPathPattern=failing-test

# 2. Check if it's a test issue or code issue
npm start  # Does app work?

# 3. If app works but test fails, fix test
# 4. If app doesn't work, rollback
git reset --hard HEAD~1
```

### If Features Break
1. **Identify** what broke (create issue)
2. **Isolate** the breaking commit (git bisect)
3. **Revert** specific commit or rollback
4. **Fix** in new branch
5. **Test** thoroughly before re-merging

## Testing Strategy

### Automated Tests
```bash
# Run before every commit
npm test

# Run before every push
npm run test:coverage

# Ensure coverage doesn't drop
# Current: ~70%
# Target: Don't drop below 65% during refactor
```

### Manual Smoke Test (5 minutes)
1. **App Startup**
   - [ ] App opens without errors
   - [ ] Welcome screen displays
   - [ ] Navigation sidebar works

2. **Project Creation**
   - [ ] Open wizard
   - [ ] Fill basic info
   - [ ] Complete triage
   - [ ] Create project successfully
   - [ ] Project appears in list

3. **Project Management**
   - [ ] Open project list
   - [ ] Search/filter works
   - [ ] Open existing project
   - [ ] Edit project details
   - [ ] Save changes

4. **Settings**
   - [ ] Open settings
   - [ ] Change configuration
   - [ ] Save settings
   - [ ] Settings persist

5. **Agency Directory**
   - [ ] Open directory
   - [ ] Search agencies
   - [ ] View agency details

### Critical Path Testing
Before any major merge, test these scenarios:

1. **New User Onboarding**
   - Clear localStorage
   - Fresh start
   - Create first project

2. **Existing User**
   - Load app with existing data
   - All projects visible
   - Can create new project

3. **Revision Workflow**
   - Create revision project
   - File copying works
   - Validation works

4. **Workload Dashboard**
   - Load users
   - Create assignment
   - View workload

## Rollback Triggers

Immediately rollback if:
- ❌ App won't start
- ❌ Data loss occurs
- ❌ Critical feature breaks
- ❌ Tests fail with no clear fix
- ❌ Build process breaks
- ❌ Performance degrades >20%

Consider rollback if:
- ⚠️ Subtle bugs introduced
- ⚠️ User experience degrades
- ⚠️ Code becomes harder to understand
- ⚠️ Technical debt increases
