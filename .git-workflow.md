# Git Workflow for Safe Refactoring

## Branch Strategy

```
main (protected)
  ├── develop (current working version)
  │   ├── refactor/phase1-ipc-handlers
  │   ├── refactor/phase1-service-consolidation
  │   ├── refactor/phase2-react-performance
  │   └── refactor/phase3-typescript
  └── release/v5.1.0
```

## Safety Rules

1. **Never commit directly to main**
2. **Always create feature branches**
3. **Merge only after tests pass**
4. **Keep commits small and atomic**
5. **Write descriptive commit messages**

## Pre-Commit Checklist

- [ ] All tests pass
- [ ] No console errors
- [ ] App starts successfully
- [ ] Core features tested manually
- [ ] No linter errors
- [ ] Code reviewed

## Emergency Rollback

If something breaks:
```bash
# Quick rollback to last known good state
git reset --hard HEAD~1

# Or revert specific commit
git revert <commit-hash>

# Or switch to backup branch
git checkout main
```

## Commit Message Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

Types:
- `refactor`: Code restructuring without behavior change
- `perf`: Performance improvements
- `test`: Adding tests
- `docs`: Documentation
- `chore`: Maintenance tasks

Example:
```
refactor(ipc): Extract project handlers to separate module

- Moved 30 project-related IPC handlers to handlers/project.js
- No behavior changes, only code organization
- All existing tests pass

Relates-to: #REFACTOR-PHASE1
```
